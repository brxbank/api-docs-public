# MED — Mecanismo Especial de Devolução

MED (Special Return Mechanism) is the BACEN process for returning PIX funds in cases of fraud or operational errors. Your system may be involved in two roles:

- **Inbound MED** — BACEN requests a return of funds you received (you are the accused party)
- **Outbound MED** — You initiate a return request for funds you sent and want back (you are the claimant)

<Callout intent="info">
  **Scopes required:** `pix:med:read` (list, get) and `pix:med:write` (create, respond, cancel).
  Infraction endpoints require `pix:infractions:read` and `pix:infractions:write`.
</Callout>

---

## Inbound MED Journey

This is the flow when BACEN opens a MED against your account — you are being asked to return funds.

```
BACEN/JDPI
    │
    ▼ (1) MED refund request arrives
gw-pix-service ──► pix.med_refund.received webhook ──► your system
    │
    ▼
  status: OPEN
    │
    ├── (2) You submit a defense (approve or reject)
    │       POST /api/v1/pix/med/{id}/respond
    │       action: "approve" | "reject", reason: "..."
    │
    ▼
  status: UNDER_REVIEW
    │
    ▼ pix.med_refund.defense_submitted webhook
    │
    ▼ (3) BACEN resolves
pix.med_refund.resolved webhook
  status: CANCELLED (rejected/not returned) or COMPLETED (returned)
```

### Step 1 — Receive the webhook

When a MED arrives, your registered endpoint receives a `pix.med_refund.received` event:

```json
{
  "id": "d4e5f6a7-b8c9-0d1e-2f3a-4b5c6d7e8f9a",
  "source": "pixgw",
  "event": "pix.med_refund.received",
  "timestamp": "2026-07-23T14:30:00Z",
  "data": {
    "refund_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "external_refund_id": "JDPI-REF-00123456",
    "end_to_end_id": "E1234567820260723123456789abcdef",
    "e2e": "E1234567820260723123456789abcdef",
    "amount": "1500.00",
    "refund_reason": "FRAUD",
    "direction": "INBOUND",
    "status": "OPEN",
    "hold_placed": false,
    "data_recebimento": "23/07/2026",
    "hora_recebimento_med": "11:30:00",
    "bancocontestante": "Banco Bradesco S.A."
  }
}
```

**`refund_reason` values:** `FRAUD` | `OPERATIONAL_FLAW` | `UNDISCLOSED`

### Step 2 — Fetch the MED details

```bash
curl -X GET "https://api.brzip.com.br/api/v1/pix/med/{refund_id}" \
  -H "Authorization: Bearer <your-jwt>"
```

**Response:**

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "external_refund_id": "JDPI-REF-00123456",
  "end_to_end_id": "E1234567820260723123456789abcdef",
  "client_id": "your-client-uuid",
  "amount": "1500.00",
  "refund_reason": 1,
  "status": "OPEN",
  "direction": "INBOUND",
  "defense_reason": null,
  "defense_submitted_at": null,
  "created_at": "2026-07-23T14:30:00Z",
  "updated_at": "2026-07-23T14:30:00Z"
}
```

**`refund_reason` as integer:** `1` = FRAUD | `2` = OPERATIONAL_FLAW | `3` = UNDISCLOSED

### Step 3 — Respond (approve or reject the return)

Call the respond endpoint. For inbound MEDs, `action` must be `"approve"` or `"reject"` and `reason` is required:

```bash
# Approve — agree to return the funds
curl -X POST "https://api.brzip.com.br/api/v1/pix/med/{refund_id}/respond" \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "approve",
    "reason": "Transaction confirmed as fraudulent after internal review"
  }'
```

```bash
# Reject — dispute the MED request
curl -X POST "https://api.brzip.com.br/api/v1/pix/med/{refund_id}/respond" \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "reject",
    "reason": "Transaction was authorized by account holder — legitimate purchase confirmed"
  }'
```

**Response:**

```json
{
  "message": "defense submitted",
  "refund_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "UNDER_REVIEW"
}
```

After this call, a `pix.med_refund.defense_submitted` webhook fires and the status moves to `UNDER_REVIEW`.

### Step 4 — Await resolution

BACEN analyzes the defense and sends a final decision. You receive `pix.med_refund.resolved`:

```json
{
  "id": "f7a8b9c0-d1e2-3f4a-5b6c-7d8e9f0a1b2c",
  "source": "pixgw",
  "event": "pix.med_refund.resolved",
  "timestamp": "2026-07-25T09:00:00Z",
  "data": {
    "refund_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "external_refund_id": "JDPI-REF-00123456",
    "end_to_end_id": "E1234567820260723123456789abcdef",
    "e2e": "E1234567820260723123456789abcdef",
    "amount": "1500.00",
    "refund_reason": "FRAUD",
    "direction": "INBOUND",
    "status": "CANCELLED",
    "resolved_by": "bacen_event",
    "data_recebimento": "25/07/2026",
    "hora_recebimento_med": "06:00:00",
    "bancocontestante": "Banco Bradesco S.A."
  }
}
```

**Final `status` values:**
- `COMPLETED` — funds were returned to the claimant
- `CANCELLED` — MED was rejected/dropped; funds remain with you

## Outbound MED Journey

This is the flow when **you** initiate a MED — you sent PIX funds and want them back (fraud or operational error).

```
your system
    │
    ▼ (1) POST /api/v1/pix/med
        direction: "OUTBOUND"
    │
    ▼
  status: OPEN → workflow started (JDPI opened)
    │
    ▼ (2) Counterparty responds via JDPI (days or weeks)
    │
    ▼ (3) You confirm outcome
        POST /api/v1/pix/med/{id}/respond
        action: "completed" | "cancelled"
    │
    ▼
pix.med_refund.resolved (status: COMPLETED or CANCELLED)
```

### Step 1 — Open an outbound MED

```bash
curl -X POST "https://api.brzip.com.br/api/v1/pix/med" \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "end_to_end_id": "E1234567820260710143022abcdef1234",
    "amount": "850.00",
    "refund_reason": 1,
    "direction": "OUTBOUND",
    "details": "Recipient unrecognized — possible fraud"
  }'
```

**Request fields:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `end_to_end_id` | string | Yes | E2E ID of the original PIX transaction |
| `amount` | string | Yes | Amount to recover (in BRL, decimal string) |
| `refund_reason` | integer | Yes | `1` = FRAUD, `2` = OPERATIONAL_FLAW, `3` = UNDISCLOSED |
| `direction` | string | Yes | `"OUTBOUND"` |
| `details` | string | No | Free-text description |

**Response (201):**

```json
{
  "id": "c3d4e5f6-a7b8-9c0d-ef12-34567890abcd",
  "external_refund_id": null,
  "end_to_end_id": "E1234567820260710143022abcdef1234",
  "client_id": "your-client-uuid",
  "amount": "850.00",
  "refund_reason": 1,
  "status": "OPEN",
  "direction": "OUTBOUND",
  "created_at": "2026-07-23T15:00:00Z",
  "updated_at": "2026-07-23T15:00:00Z"
}
```

### Step 2 — Confirm outcome (when BACEN resolves)

Once the counterparty responds through JDPI, confirm the outcome:

```bash
# Funds returned successfully
curl -X POST "https://api.brzip.com.br/api/v1/pix/med/{refund_id}/respond" \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{"action": "completed"}'

# Return not obtained
curl -X POST "https://api.brzip.com.br/api/v1/pix/med/{refund_id}/respond" \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{"action": "cancelled"}'
```

## MED Statuses

| Status | Description |
|--------|-------------|
| `OPEN` | MED received or created — awaiting response |
| `WAITING_RETURN` | Counterparty acknowledged — awaiting fund transfer |
| `UNDER_REVIEW` | Defense submitted — BACEN is analyzing |
| `COMPLETED` | Funds returned successfully (final) |
| `CANCELLED` | MED closed without return (rejected or dropped) (final) |

## Infractions

PIX infractions (`/infractions`) are formal reports to BACEN about suspicious transactions — separate from MED but closely related. Infractions trigger BACEN investigations; MED is the mechanism for the actual fund return.

### Creating an outbound infraction report

```bash
curl -X POST "https://api.brzip.com.br/api/v1/pix/infractions" \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "end_to_end_id": "E1234567820260710143022abcdef1234",
    "report_type": "FRAUD",
    "reporter_ispb": "12345678",
    "direction": "OUTBOUND",
    "details": "Suspected fraudulent recipient account"
  }'
```

**Request fields:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `end_to_end_id` | string | No* | E2E ID of the flagged transaction |
| `pix_key_id` | string | No* | PIX key involved (alternative to E2E) |
| `report_type` | string | Yes | `FRAUD` \| `AML_CFT` \| `ACCOUNT_CLOSURE` \| `OTHER` |
| `reporter_ispb` | string | Yes | ISPB of the reporting institution |
| `reported_ispb` | string | No | ISPB of the reported institution |
| `direction` | string | Yes | `INBOUND` \| `OUTBOUND` |
| `details` | string | No | Free-text description |

*At least one of `end_to_end_id` or `pix_key_id` is expected.

**Response (201):**

```json
{
  "id": "inf-uuid-here",
  "external_report_id": null,
  "end_to_end_id": "E1234567820260710143022abcdef1234",
  "report_type": "FRAUD",
  "reporter_ispb": "12345678",
  "status": "OPEN",
  "direction": "OUTBOUND",
  "created_at": "2026-07-23T15:00:00Z",
  "updated_at": "2026-07-23T15:00:00Z"
}
```

### Infraction statuses

| Status | Description |
|--------|-------------|
| `OPEN` | Report created — under analysis |
| `ACKNOWLEDGED` | Counterparty acknowledged the report |
| `CANCELLED` | Report cancelled |
| `CLOSED` | Investigation concluded (final) |

## All MED Webhook Events

Subscribe to these events on your webhook to cover the full MED lifecycle:

```bash
curl -X POST "https://api.brzip.com.br/api/v1/webhooks" \
  -H "Authorization: Bearer <your-jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://your-system.com/webhooks/med",
    "events": [
      "pix.med_refund.received",
      "pix.med_refund.defense_submitted",
      "pix.med_refund.resolved",
      "pix.infraction.received",
      "pix.infraction_notification.received"
    ]
  }'
```

| Event | When it fires |
|-------|--------------|
| `pix.med_refund.received` | BACEN opens an inbound MED against you |
| `pix.med_refund.defense_submitted` | Your defense was recorded (status → UNDER_REVIEW) |
| `pix.med_refund.resolved` | BACEN issued a final decision (COMPLETED or CANCELLED) |
| `pix.infraction.received` | BACEN sent an inbound infraction report |
| `pix.infraction_notification.received` | JDPI §8.2.21 infraction notification received |

All events arrive as an HTTP POST to your URL with this envelope:

```json
{
  "id": "d4e5f6a7-b8c9-0d1e-2f3a-4b5c6d7e8f9a",
  "source": "pixgw",
  "event": "pix.med_refund.received",
  "timestamp": "2026-07-23T14:30:00Z",
  "data": { }
}
```

| Envelope field | Type | Description |
|----------------|------|-------------|
| `id` | string (UUID) | Unique delivery ID |
| `source` | string | Always `pixgw` |
| `event` | string | Event name (see table above) |
| `timestamp` | string | ISO 8601 UTC timestamp |
| `data` | object | Event-specific payload — detailed per event below |
